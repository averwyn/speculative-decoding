# 第1章 绪论
### 1.1 研究背景
近年来，随着 Transformer 架构和大规模预训练方法的快速发展，大语言模型在自然语言理解与生成任务中展现出较强能力，并在智能问答、文本摘要、代码生成和辅助写作等场景中得到广泛应用。大语言模型通常采用自回归方式逐 token 生成文本，即每一步基于已有前缀预测下一个 token，并将其追加到输入中继续生成。这种生成方式建模过程清晰、输出质量稳定，但也存在明显局限：生成过程天然串行，后一个 token 的预测依赖前一个 token 的结果，难以充分利用现代硬件的并行计算能力。当输出序列较长或模型规模较大时，逐 token 解码会带来较高推理延迟，成为制约大语言模型实际部署与应用的重要瓶颈。  

为缓解这一问题，学术界和工业界围绕大模型推理加速开展了大量研究，包括模型压缩、量化、KV cache 优化、算子融合、并行推理和新型解码方法等。其中，推测解码（Speculative Decoding）是近年来受到广泛关注的一类解码加速技术。其核心思想是引入一个计算开销较低的草稿模型，先快速提出多个候选 token，再由目标模型对这些候选序列进行并行验证，从而减少目标模型逐 token 串行解码的次数。在理论上，若草稿模型与目标模型在局部预测上具有较高一致性，则推测解码能够在不改变目标分布正确性的前提下提升生成效率。  

尽管推测解码具有较强的理论吸引力，但其实际效果并非始终理想。一方面，草稿模型与目标模型之间的分布差异会直接影响候选 token 的接受率；另一方面，候选长度设置、采样策略选择、缓存维护方式以及工程实现细节也会显著影响最终端到端性能。尤其在计算资源受限的本地单卡环境下，推测解码的真实收益常常受到草稿模型额外开销和回退恢复开销的制约。因此，围绕“何种条件下推测解码能够真正产生收益”以及“不同解码方式与模型配对如何影响接受行为和加速效果”等问题开展系统研究，具有明确的研究价值。

在此背景下，本文以自回归大模型为对象，围绕一级异构推测解码的设计、实现与实验分析展开研究，并进一步讨论多层级异构推测解码的扩展可能性，希望为资源受限条件下的大模型推理优化提供可复现的实验依据和分析参考。

### 1.2 研究意义
大语言模型的推理效率问题既是当前模型部署中的现实问题，也是模型系统优化研究中的关键问题。围绕推测解码开展研究，主要具有以下三方面意义。

首先，在理论层面，推测解码为缓解自回归生成的串行瓶颈提供了一种不同于模型压缩和硬件堆叠的优化思路。它并不直接改变目标模型本身的参数结构，而是通过“轻量草稿模型提议 + 目标模型并行验证”的方式重构生成流程。围绕接受率、平均接受前缀长度、候选长度和开销拆解展开分析，有助于从解码机制、分布一致性和系统开销之间的关系出发，深化对大模型推理效率来源的理解。

其次，在工程层面，推测解码能否在真实环境中产生端到端收益，并不单纯由理论决定，而与模型配对、采样策略、缓存管理和实现方式密切相关。尤其在本地单卡、显存和算力有限的条件下，很多理论上可行的加速方案未必能够转化为实际收益。本文在统一实验口径下比较不同模型配对和不同解码策略的表现，能够帮助识别影响推测解码效率的关键因素，为后续工程优化提供依据。

最后，在应用与拓展层面，大语言模型正不断向终端部署、本地部署和边缘推理方向扩展，有限硬件条件下的可用性与性能收益真实性愈发重要。本文的研究场景贴近资源受限环境，其结论对中小规模模型部署、教学实验平台以及个人研究环境中的推理优化具有一定参考价值。同时，一级推测解码也是多层级异构推测解码的重要基础，对一级场景中草稿成本、目标验证收益和接受行为的理解，也能为后续多层级结构设计提供依据。

### 1.3 国内外研究现状
随着大语言模型推理成本问题日益突出，国内外研究者围绕自回归生成加速提出了多种优化方法。总体来看，现有研究大致可以分为两类：一类是从模型本身入手，通过蒸馏、剪枝、量化等方式降低单次前向计算成本；另一类是从推理过程与解码机制入手，通过缓存优化、并行策略和新型解码方法减少逐 token 生成中的冗余计算。推测解码属于后者，并逐渐成为近年来推理加速研究中的热点方向之一。

在推测解码基础研究方面，Leviathan 等人提出了较为经典的 speculative decoding 框架，系统阐述了利用草稿模型与目标模型协同生成、在保证目标分布正确性的前提下减少目标模型调用次数的基本思想。这一工作奠定了推测解码的理论基础，也使“候选生成—并行验证—拒绝回退”的流程成为后续研究的重要出发点。此后，相关研究开始从不同角度对该方法进行扩展，例如提升候选接受率、降低验证成本、改进修正采样过程，以及将推测解码与不同模型结构、不同硬件环境结合起来。

在多层级与异构推测解码方向，研究者进一步尝试将单一级草稿模型扩展为多层草稿链条，即由更轻量的模型向更重的模型逐级提出候选，从而希望进一步减少最重目标模型的有效调用频率。这类方法的核心动机在于，若相邻层之间都能保持较高接受率，则理论上可逐级放大推理加速收益。近年来出现的一些工作已开始关注多层级 speculative decoding、量化草稿模型、多级验证链条等问题，显示出该方向具有持续演进潜力。但与此同时，多层级结构也带来了更复杂的状态维护、更长的 proposal 链条以及更高的工程实现难度，其实际效果仍受到层间匹配关系和系统开销的共同制约。

除传统草稿—目标协同框架外，一些研究还尝试通过训练阶段增强模型在推理阶段的推测能力，例如让模型显式学习未来 token 结构或中间表示，以提高推测生成质量。这类方法在一定程度上打通了训练与推理之间的界限，为推测解码提供了新的研究思路。

国内关于大语言模型推理优化的研究也在持续推进，重点多集中于推理框架优化、KV cache 管理、低比特量化部署以及面向中文场景的大模型推理效率提升。相较而言，围绕 speculative decoding 尤其是“异构模型配对”“不同采样方式对接受行为的影响”“本地资源受限环境下的端到端性能表现”等细粒度问题的系统实证研究仍相对有限。许多已有工作更关注理论可行性或服务器级部署收益，而对于本地单机条件下不同模型组合是否真正有效、何种设置更适合实际使用，尚缺少细致分析。

总体而言，国内外已有研究为推测解码提供了坚实基础，但仍存在以下不足：其一，不同模型配对与不同候选长度对实际收益的影响规律仍有待细化；其二，采样策略对接受行为和输出一致性的影响尚未形成统一结论；其三，多层级扩展的收益边界与实现难点仍需在一级实验基础上进一步厘清。本文正是在上述背景下，围绕一级异构推测解码的设计与实验展开分析，并将多层级异构推测解码作为进一步讨论方向。

### 1.4 本文研究问题与研究目标
围绕自回归大模型推理效率优化这一核心问题，本文重点关注以下几个研究问题。

第一，在本地单机、资源受限的实验环境下，一级异构推测解码能否在端到端生成过程中体现出稳定收益，即理论上的并行验证优势能否抵消草稿模型提议和拒绝回退带来的额外开销。

第二，不同解码方式会如何影响推测解码的接受行为和整体性能。由于贪心解码、Top-k 采样和 Top-p 采样在 token 选择机制上存在明显差异，因此它们可能改变草稿模型与目标模型的局部一致性，并进一步影响接受率、平均接受前缀长度和输出一致性。

第三，候选长度的设置应如何影响推测解码效果。候选长度过小会削弱目标模型并行验证收益，过大则可能放大候选尾部失配风险和草稿侧提议开销，因此需要识别不同配置下较优的候选长度区间。

第四，不同模型配对是否会显著改变推测解码的有效性。并不是任意轻量模型都适合作为草稿模型，只有在“草稿提议成本”和“与目标模型的分布接近度”之间达到较好平衡时，推测解码才可能体现优势。

第五，在一级异构推测解码实验基础上，多层级异构推测解码是否具有进一步扩展的潜力，以及其主要难点体现在哪些方面。

基于上述问题，本文的研究目标可以概括为：设计并实现一个可复现的一级异构推测解码实验框架；在统一实验口径下比较不同解码方式、不同候选长度和不同模型配对的表现；从接受行为与端到端开销两方面解释推测解码性能变化原因；最后结合实验结果分析多层级异构推测解码的潜在优势与实现挑战。

### 1.5 本文主要研究内容
围绕上述研究目标，本文主要开展了以下几个方面的工作。

第一，系统梳理了大语言模型自回归生成机制、常见解码方法和推测解码基本原理，分析了贪心解码、Top-k 采样、Top-p 采样以及候选接受、拒绝回退和并行验证等关键机制，为后续方法设计奠定理论基础。

第四，在一级实验基础上，对系统性能瓶颈和多层级异构推测解码扩展方向进行了初步讨论，以揭示瓶颈迁移规律，并为后续研究提供思路。

# 第2章 相关理论与关键技术
### 2.1 大语言模型的自回归生成机制
大语言模型通常采用因果语言建模（Causal Language Modeling）目标进行训练，其核心任务是根据已有前缀预测下一个 token 的概率分布。设输入序列为

$ x=(x_1​,x_2​,…,x_n​), $

则模型对整个序列的联合概率可写为  

$ P(x)=\prod_{t=1}^{n}P(x_t\mid x_{<t})， $

其中$ x_{<t} $表示位置$ t $之前的所有 token。由此可见，模型的生成过程本质上是按时间步递推展开的：在第$ t $步，模型基于当前前缀$ x_{<t} $计算下一 token 的条件分布，再根据某种解码策略选择一个 token 作为输出，并将其拼接到前缀后继续进行下一步预测。

这种自回归生成机制使模型能够自然处理开放式文本生成任务，且在表达流畅性和上下文一致性方面具有较强优势。然而，自回归生成也带来明显的串行依赖：第$ t+1 $个 token 的计算必须依赖第$ t $个 token 的结果，因此整个生成过程难以像训练阶段那样对多个位置完全并行处理。这种串行性在模型规模不断增大、输出长度不断增加的背景下，会显著放大推理延迟。

在 Transformer 架构中，为了避免每一步都重新计算全部历史前缀，推理阶段通常会引入 KV cache 机制，将历史 token 在各层注意力中的 key 和 value 表示缓存下来。这样在生成新 token 时，只需对新增位置执行前向计算，并与历史缓存进行拼接即可。KV cache 可以有效降低重复计算，但并不能改变逐 token 串行生成的本质，因此在长文本生成或大模型推理中，解码效率问题仍然突出。

从系统角度看，自回归生成的总耗时不仅与模型单次前向的开销有关，还与解码轮数密切相关。若一个长度为$ m $的输出需要执行$ m $次目标模型前向，则任何能够减少“有效解码轮数”的方法，都有可能带来性能收益。推测解码正是在这一背景下提出：它试图通过草稿模型一次提出多个候选 token，再由目标模型对整段候选进行验证，从而减少目标模型逐 token 串行解码的次数。

因此，理解大语言模型的自回归生成机制，是分析推测解码为何可能有效、又为何容易受到草稿分布偏差和系统开销限制的基础。

### 2.2 常见解码方法
在大语言模型生成过程中，模型输出的是下一 token 的概率分布，而真正生成哪个 token，还需要由解码方法决定。不同解码方法会改变输出文本的确定性、随机性、多样性以及局部稳定性，也会进一步影响推测解码中草稿模型与目标模型之间的接受行为。本文主要关注贪心解码、Top-k 采样和 Top-p 采样三种常见方法。  

#### 2.2.1 贪心解码
贪心解码（Greedy Decoding）是在每一步直接选择当前条件概率最大的 token 作为输出，即

$ x_t=\arg\max_{v\in V}P(v\mid x_{<t}) $

其中$ V $表示词表集合。该方法的特点是实现简单、输出稳定、可重复性强。在相同输入和相同模型参数下，贪心解码总会得到确定的输出结果。

贪心解码的优势在于局部最优选择明确，适合追求确定性结果的场景，例如某些信息抽取、固定格式生成或需要严格可复现的实验环境。同时，由于每一步只关心概率最大的 token，草稿模型与目标模型在贪心路径上的一致性通常更容易观察，因此它也是分析推测解码接受行为时最基础的一种设置。

不过，贪心解码也存在局限。它只关注当前步的最大概率 token，缺乏对后续整体文本质量的全局考虑，容易导致表达单一、缺少多样性，甚至在某些开放式生成任务中陷入局部高概率但整体质量一般的输出路径。因此在实际应用中，采样类解码方法往往更常见。

#### 2.2.2 Top-k 采样
Top-k 采样（Top-k Sampling）是一种在保留一定随机性的同时限制候选空间的方法。其基本思想是：在当前时刻先选取概率最高的前$ k $个 token 构成候选集合，然后仅在该集合内按归一化后的概率分布进行采样。设排序后保留下来的集合为$ V_k $，则实际采样分布为

$ P'(v\mid x_{<t})=
\begin{cases}
\dfrac{P(v\mid x_{<t})}{\sum\limits_{u\in V_k}P(u\mid x_{<t})}, & v\in V_k,\\[8pt]
0, & v\notin V_k.
\end{cases}
 $

Top-k 采样相较于纯随机采样更加可控，因为低概率 token 会被直接屏蔽，不参与最终选择。这样既能避免完全确定性的输出，又能减少生成异常 token 的风险。当$ k $较小时，生成结果更接近贪心；当 $ k $较大时，随机性和多样性会进一步增强。

在推测解码场景中，Top-k 采样会使草稿模型每一步生成的候选 token 不再唯一，因而候选序列与目标模型的局部一致性更容易受到随机因素影响。也正因为如此，Top-k 采样是考察推测解码在非确定性生成条件下表现的重要设置。

#### 2.2.3 Top-p 采样
Top-p 采样（Top-p Sampling），又称 nucleus sampling，其思想不是固定保留前$ k $个 token，而是动态选择一个最小候选集合，使该集合中的累计概率达到给定阈值$ p $。设排序后的 token 概率依次为$ p_1,p_2,…, $则选择最小的$ m $，满足

$ 
\sum_{i=1}^{m}p_i \ge p
 $

再仅在这$ m $个 token 构成的集合中进行归一化采样。

Top-p 采样相较于 Top-k 的主要特点是候选集合大小可随分布形态动态变化。当模型当前分布较为尖锐时，达到阈值$ p $所需的 token 数较少；当分布较为平缓时，候选集合会自动扩大。因此，Top-p 采样通常被认为在保证生成多样性和避免低质量长尾 token 之间具有更好的自适应性。

在推测解码中，Top-p 采样带来的动态截断会进一步影响草稿分布与目标分布之间的关系。一方面，它可能更贴近真实文本生成场景；另一方面，也会使候选分布的形态随上下文而显著变化，从而使接受率和平均接受前缀长度的变化更加复杂。因此，Top-p 采样也是本文重点考察的一类解码方

### 2.3 推测解码基本原理
推测解码是一种面向自回归大语言模型的推理加速方法，其核心思想是在保持目标模型输出分布正确性的前提下，尽可能减少目标模型逐 token 串行解码的次数。为此，推测解码引入一个计算成本较低的草稿模型，由其先快速提出若干候选 token，再由目标模型利用已有前缀缓存对这些候选进行并行验证。若候选被接受，则多个 token 可以一次性并入最终输出；若候选中途被拒绝，则在拒绝位置进行修正采样并恢复状态，然后继续下一轮生成。  

#### 2.3.1 一级推测解码流程
一级推测解码是最基本的推测解码形式，仅包含一个草稿模型和一个目标模型。其基本流程可概括如下：

（1）给定输入 prompt，先由草稿模型和目标模型分别完成前缀预填充，并建立各自的缓存状态；  
（2）在当前生成轮次中，草稿模型基于已有前缀连续提出长度为$ k $的候选 token 序列；  
（3）目标模型利用当前前缀缓存对整段候选序列进行一次前向验证，得到对应位置上的条件分布；  
（4）系统从前到后检查候选 token 是否满足接受条件，得到该轮被接受的连续前缀；  
（5）若整段候选被接受，则目标模型在候选末端再生成一个真实 token，作为本轮额外推进；若在某一位置首次拒绝，则根据目标分布与草稿分布构造修正分布，在拒绝位置采样新的 token，并执行状态回退；  
（6）重复以上过程，直到达到预设最大生成长度。

这一流程的关键在于，目标模型并不是对每个 token 单独执行一次前向传播，而是借助候选块级验证一次性获得多个位置的条件分布，从而减少高成本模型的调用频率。

#### 2.3.2 接受与回退机制
推测解码能否成立，关键在于如何在保证目标分布正确性的前提下接受草稿候选。对于候选序列中某个位置的 token $ x $，若草稿模型给出的概率为 $ q(x) $，目标模型给出的概率为 $ p(x) $，则在采样设置下通常按照

$ \alpha(x)=\min\left(1,\frac{p(x)}{q(x)}\right)
 $

的规则决定是否接受该 token。若被接受，则该 token 可以直接并入最终输出；若拒绝，则系统在该位置停止继续检查后续候选，并根据目标分布与草稿分布之间的差异构造修正分布，从中采样新的 token 作为真实输出延续。

在贪心解码条件下，接受判定可以进一步简化：若目标模型当前分布下的最大概率 token 与草稿模型提出的 token 一致，则视为接受；否则视为拒绝。由于贪心路径本身是确定性的，因此这种判定方式在实验中更容易分析。

一旦发生拒绝，推测解码就必须处理状态回退问题。由于草稿模型和目标模型都可能已经沿着候选路径更新了缓存状态，因此拒绝后需要将缓存裁剪到已接受前缀结束的位置，再接入修正 token 继续推进。状态回退处理是否高效，会直接影响推测解码的端到端性能。

#### 2.3.3 推测解码的潜在加速来源
推测解码的加速潜力主要来源于以下两个方面。

第一，减少高成本目标模型的有效解码轮数。在普通自回归生成中，目标模型需要对每个新增 token 单独执行一次下一步预测；而在推测解码中，若一轮内有多个候选 token 被接受，则目标模型一次前向传播可以“覆盖”多个输出位置，从而摊薄逐 token 解码成本。

第二，利用较轻量草稿模型承担提议工作。若草稿模型远轻于目标模型，且局部预测与目标模型足够接近，那么候选生成的额外开销相对较小，而接受收益较大，整体上就可能带来加速效果。

不过，推测解码并不保证一定加速。若草稿模型与目标模型差异较大，则接受率会下降，拒绝回退会增多；若草稿模型本身并不轻量，则额外的 proposal 成本也会抵消目标模型验证节省下来的收益。因此，推测解码的实际效果始终取决于“提议成本—分布接近度—验证收益”之间的综合平衡。

### 2.4 推测解码的关键影响因素
推测解码的性能并非由某一个单独因素决定，而是由模型关系、候选长度、解码方式和工程实现等多个变量共同作用的结果。理解这些关键影响因素，有助于解释为什么同一套推测解码框架在不同实验配置下会呈现出显著不同的端到端效果。  

#### 2.4.1 草稿模型与目标模型的匹配关系
草稿模型与目标模型的匹配关系是影响推测解码效果的核心因素之一。理想情况下，草稿模型应同时满足两个条件：一是计算开销明显低于目标模型，能够快速提出候选；二是与目标模型在局部预测上保持较高一致性，从而提高接受率。

若草稿模型过于轻量，其推理速度虽然较快，但可能因为建模能力不足而频繁提出错误候选，导致接受率偏低、回退次数增多。反之，若草稿模型过于接近目标模型，虽然局部一致性可能较高，但自身前向成本也会上升，从而削弱“快速提议”的价值。因此，推测解码并不是单纯选择越小的模型越好，而是需要在“足够便宜”和“足够接近”之间取得平衡。  

#### 2.4.2 候选长度对接受率与效率的影响
候选长度是指草稿模型在每一轮中连续提出的候选 token 数。设候选长度为$ k $，则当$ k $较小时，目标模型一次验证所能覆盖的 token 数较少，并行收益有限；当$ k $较大时，理论上单轮可以推进更多 token，从而减少目标模型的有效调用次数。

然而，随着$ k $增大，候选块后部位置更容易出现分布偏移，导致接受率下降和回退开销上升。同时，草稿模型也需要逐 token 提出更长的候选序列，其前向开销会近似随$ k $线性增长。因此，候选长度通常存在一个较优区间，而不是越大越好。

#### 2.4.3 解码方式对候选质量的影响
不同解码方式会直接改变草稿模型提出候选序列的方式，也会进一步影响目标模型的接受行为。贪心解码路径确定性强，草稿模型与目标模型只需在最大概率 token 上保持一致，因此通常更容易形成较长的接受前缀。相比之下，Top-k 和 Top-p 等采样策略引入随机性，使候选路径更灵活，但也更容易在较早位置产生失配。

因此，在推测解码研究中，解码方式不仅是输出风格控制参数，也是影响性能的重要变量。不同解码方式下的接受率、平均接受前缀长度和输出一致性都可能显著不同。

#### 2.4.4 系统实现开销对端到端性能的影响
除了算法层面的接受行为之外，系统实现开销同样会显著影响推测解码的真实收益。具体而言，草稿模型提议时间、目标模型验证时间、拒绝后的缓存恢复时间，以及不同框架下的实现快路径差异，都会体现在端到端吞吐率和总耗时之中。

例如，在某些实现中，即使接受率看起来较高，但若缓存裁剪和状态恢复代价过大，整体性能仍可能不如纯目标模型自回归生成。同样，若草稿模型本身前向开销已接近目标模型，proposal 成本也会快速吞噬理论收益。因此，对推测解码进行开销拆解分析，是理解其真实性能边界的重要手段。

### 2.5 多层级推测解码概述
在一级推测解码中，系统只包含一个草稿模型和一个目标模型。若进一步考虑多个不同规模的模型按层次组织起来，就形成了多层级推测解码。该方法希望通过更细粒度的层间提议与验证关系，进一步降低最重目标模型的调用频率，从而放大推理加速效果。  

#### 2.5.1 多层级推测解码的基本思想
多层级推测解码的基本思路是：不再由单个草稿模型直接向目标模型提出候选，而是构建一个由轻到重的模型链条。最轻量的模型先提出初始候选，随后由中间层模型对这些候选进行进一步验证或再提议，最后再由最重的目标模型执行最终验证。若各层之间的预测具有较强一致性，则理论上可以显著减少最高成本模型的直接解码轮数。

从结构上看，多层级方法可以被理解为把一级推测解码中的“草稿—目标”关系拓展为“草稿1—草稿2—…—目标”的层次化关系。其核心仍是利用便宜模型承担尽可能多的提议工作，再让更重模型负责纠偏和保证分布正确性。

#### 2.5.2 多层级推测解码的潜在优势
多层级推测解码的潜在优势主要体现在两个方面。

一方面，它可能进一步减少最重目标模型的有效调用次数。若一级推测解码已经能够让目标模型一次前向覆盖多个 token，那么在多层级结构下，中间层模型还可能进一步吸收一部分验证压力，使最重模型只在更少轮次中参与最终纠偏。

另一方面，多层级结构为不同规模模型之间的协同提供了更灵活的空间。对于模型规模差距较大的场景，单一级草稿模型可能既不够快也不够接近目标模型，而引入中间层模型后，层间分布差距有望被逐级缩小，从而提高整体接受效率。

#### 2.5.3 多层级推测解码的实现难点
尽管多层级推测解码在理论上具有吸引力，但其实现难度也明显高于一级结构。

首先，层数增加意味着状态维护更加复杂。每一层模型都需要维护自己的缓存状态，一旦下游层发生拒绝，上游多层缓存都可能需要同步回退和重建，这会显著增加系统实现复杂度与时间开销。

其次，proposal 链条变长后，前端模型累计成本可能快速增长。即便最高层目标模型的调用减少了，如果前几层草稿模型本身已经消耗了大量时间，那么端到端收益未必成立。

再次，多层级结构更依赖相邻层之间的匹配质量。若某一层与上一层或下一层之间分布差异过大，则接受率会迅速恶化，导致整个链条中断，甚至比一级结构更低效。

因此，多层级推测解码是否真正有效，并不取决于层数本身，而取决于相邻层模型是否能够同时满足“足够轻量、足够接近、额外开销可控”这几个条件。也正因为如此，一级推测解码的实证研究仍然是多层级扩展的重要前提。

# 第3章 一级异构推测解码方法设计与实现
### 3.1 研究思路与总体框架
本文围绕一级异构推测解码展开方法设计与系统实现。所谓“一级异构”，是指在同一轮生成过程中仅设置一个草稿模型和一个目标模型，两者在参数规模、计算能力或模型结构上存在差异，其中草稿模型负责快速提出候选 token 序列，目标模型负责对候选序列进行并行验证，并在出现拒绝时执行纠正采样与状态回退。与传统自回归解码逐 token 串行生成不同，该方法试图通过“先快提议、后并行验证”的方式降低单位输出 token 的平均生成开销，从而提升整体生成效率。

本文的总体实现框架由四个部分组成。第一部分为模型加载与运行时环境管理模块，负责加载草稿模型、目标模型及其对应 tokenizer，并统一缓存目录、设备和本地模型读取策略。第二部分为一级推测解码核心模块，负责候选生成、并行验证、接受判定、拒绝回退和缓存状态维护。第三部分为 baseline 对照模块，用于以相同采样策略调用目标模型单独完成生成，作为推测解码效果比较基准。当前实现中 baseline 进一步区分为两类：一类是基于 Hugging Face `generate()` 接口的工程化 baseline，另一类是与推测解码采用相近控制流程的手写自回归 baseline。第四部分为实验驱动与结果汇总模块，用于组织不同 prompt、不同解码方式、不同候选长度和不同模型配对的实验，并将原始结果与汇总结果写入 CSV 文件，便于后续统计分析与绘图。

从执行逻辑上看，系统首先对输入 prompt 完成预填充，分别建立草稿模型与目标模型的初始缓存状态；随后在每一轮生成中，由草稿模型连续生成长度为 $ k $ 的候选 token 序列，再由目标模型利用已有缓存对整段候选序列进行一次性验证；若候选前缀被接受，则将这些 token 直接写入最终输出，并由目标模型额外采样一个后续 token 作为该轮结束位置；若在某一位置发生拒绝，则根据目标分布与草稿分布构造修正分布，从中采样新的 token，同时执行缓存回退与重建。上述过程循环进行，直到达到最大生成长度。

### 3.2 一级异构推测解码流程设计
#### 3.2.1 草稿候选生成过程
草稿候选生成过程对应实现中的 `propose_block`。在该过程中，系统以当前已生成前缀为条件，使用草稿模型连续生成长度为 $ k $ 的候选 token。为了兼容不同解码方式，候选生成并不是直接使用模型原始 logits，而是先对 logits 做统一归一化处理：若采用贪心解码，则仅保留最大概率位置；若采用 Top-k 采样，则仅保留概率最高的前 $ k $ 个候选；若采用 Top-p 采样，则保留累积概率不超过阈值 $ p $ 的最小候选集合。在此基础上得到规范化概率分布，再进行 token 选择。

在连续提出候选的过程中，草稿模型每生成一个 token 都会更新自身缓存状态；在采样解码条件下，系统还会记录当前步的候选概率分布，以便后续目标模型进行接受判定时使用草稿分布 $ q(x) $。在当前优化实现中，系统不再为每个候选位置额外保存完整缓存快照，而是在出现拒绝时直接对已有缓存进行裁剪并继续推进，从而减少缓存复制带来的额外开销。由此可以看出，草稿候选生成不仅承担“提出 token”的功能，也承担为后续验证和回退提供上下文支撑的作用。

#### 3.2.2 目标模型并行验证过程
目标模型并行验证过程对应实现中的 `verify_block`。与传统自回归方式逐 token 执行前向计算不同，本文实现将整段候选 token 一次性送入目标模型，并结合已有前缀缓存得到候选序列上各位置的输出 logits。这样做的核心目的，是利用目标模型一次前向传播同时获得多个候选位置的条件分布，从而降低验证阶段的重复计算。

对于候选序列中的第一个 token，其验证依据来自目标模型在当前前缀上的下一 token 分布；对于第 $ i $ 个候选 token（$ i>1 $），其验证依据来自目标模型在候选序列前 $ i-1 $ 个 token 条件下得到的分布。于是，系统可以按顺序检查每个候选 token 是否被接受。若采用贪心解码，则当目标分布的最大概率 token 与草稿 token 一致时接受；若采用采样解码，则根据 $ \min(1, p(x)/q(x)) $ 的规则进行随机接受，其中 $ p(x) $ 表示目标分布在候选 token 上的概率，$ q(x) $ 表示草稿分布在同一 token 上的概率。

这种设计保留了推测解码“目标模型保证分布正确性”的核心思想，同时尽可能利用候选块级验证减少目标模型调用次数，是整个系统效率提升的关键。

【插图建议：可在此处配“目标模型并行验证示意图”，重点展示单轮中多个候选位置如何在一次前向中完成验证。】

#### 3.2.3 候选接受与拒绝回退过程
在验证阶段，如果候选 token 从前到后连续满足接受条件，则这些 token 构成一个被接受前缀。系统将这一前缀直接写入最终生成结果，并累计接受 token 数、接受前缀长度等统计量。若整段候选均被接受，则目标模型再基于候选块末端位置的分布补采样一个 token，作为该轮验证完成后的新增输出。

如果在某个候选位置首次发生拒绝，则系统不再继续检查后续候选，而是在该位置执行纠正采样。具体做法是根据目标分布 $ p $ 与草稿分布 $ q $ 构造修正分布，即对 $ p-q $ 的正部分进行归一化；当该差值近似为零时，则退化为直接使用目标分布。随后系统从修正分布中采样得到新的 token，并将其作为本轮真实输出的延续位置。此步骤对应实现中的 `corrective_distribution`。

拒绝发生后，缓存状态需要同步调整。在当前优化实现中，目标模型侧不再对整段真实前缀进行完全重建，而是先将验证阶段得到的缓存裁剪到已接受前缀长度，再接入纠正 token 继续向前推进；草稿模型侧也采用类似思路，即将已有草稿缓存裁剪到已接受前缀位置，再用纠正 token 前进一步。这样可以减少拒绝后整段重算带来的额外代价，从而降低回退阶段的时间开销。此外，当某一轮在接入已接受 token 或纠正 token 后已经达到设定的输出长度上限时，系统会直接结束生成，而不再执行仅用于下一轮准备的额外状态推进，从而进一步压缩末轮无效恢复开销。

【插图建议：可在此处配“接受与拒绝回退机制图”，突出已接受前缀、首次拒绝位置、修正采样和缓存裁剪/恢复过程。】

#### 3.2.4 状态维护与生成终止条件
为了保证多轮生成可以稳定执行，本文在实现中维护了以下几类状态信息：其一是当前已生成 token 序列及必要时可重建的完整输入表示，用于描述 prompt 与当前生成结果的拼接关系；其二是草稿模型与目标模型各自的 KV cache，用于避免每轮从头计算历史前缀；其三是目标模型当前前缀位置的下一 token logits 与草稿模型当前下一 token logits，用于减少重复前向计算；其四是统计信息，包括候选总数、接受 token 总数、拒绝 token 总数、验证轮数、累计接受前缀长度以及草稿生成、目标验证、缓存重建的时间开销。

生成过程的终止条件主要有两种。第一，当已生成 token 数达到设定的 `max_new_tokens` 时，主循环结束。第二，在每轮生成过程中，如果本轮加入最终输出后的 token 数已达到上限，则停止继续追加新 token，以保证输出长度符合实验设定。当前实现主要以最大生成长度作为统一终止条件，便于不同实验配置之间保持可比性。

### 3.3 不同解码方式下的候选生成策略
本文为了分析不同解码方式对推测解码性能与接受行为的影响，在草稿候选生成与目标验证两个阶段统一支持贪心解码、Top-k 采样和 Top-p 采样三种策略。三种策略均通过 `normalize_logits` 和 `sample_token` 两个基础函数实现。

#### 3.3.1 基于贪心解码的候选生成
在贪心解码条件下，系统每一步都选择当前概率最大的 token 作为候选输出。实现上，先将 logits 转换为仅在 argmax 位置取有效值、其余位置为负无穷的“伪 one-hot”分布，再经 softmax 得到退化概率分布。这样处理的好处在于可以将贪心解码纳入统一的概率接口，便于后续与采样策略共用验证与统计逻辑。

对于一级推测解码而言，贪心策略具有确定性强、输出一致性高的特点。当草稿模型与目标模型在局部预测上较为一致时，贪心策略通常更容易形成较长的接受前缀；但如果两者能力差异较大，则较长候选块也可能导致后部位置快速失配。

#### 3.3.2 基于 Top-k 采样的候选生成
Top-k 采样先从当前分布中筛选概率最高的前 $ k $ 个 token，再在这一有限候选集合内按归一化概率进行采样。实现中通过对 logits 执行 `torch.topk`，保留第$ k $大值以上的元素，其余位置屏蔽为负无穷，随后 softmax 归一化得到最终采样分布。

该策略通过限制候选空间，兼顾了一定随机性与可控性。对于推测解码而言，Top-k 采样可能降低草稿模型与目标模型之间的逐 token 完全一致率，但也更贴近开放式文本生成场景中的真实采样过程。因此，在实验设计中引入 Top-k 采样有助于考察推测解码在非确定性生成条件下的适用性。

#### 3.3.3 基于 Top-p 采样的候选生成
Top-p 采样也称 nucleus sampling。该方法首先按概率从高到低排序 token，再选取最小的前缀集合，使其累计概率达到阈值 $ p $，最后仅在该集合内进行归一化采样。实现中先对 logits 排序并计算 softmax 后的累积概率，再依据阈值构造掩码，将超过阈值的 token 屏蔽，最后映射回原 vocabulary 空间。

与 Top-k 相比，Top-p 的候选集合大小是动态变化的，能够更好地适应不同上下文下分布陡峭程度的变化。在推测解码场景中，这种动态候选截断可能进一步影响草稿分布与目标分布之间的相似性，因此也是需要重点考察的变量之一。

### 3.4 候选长度设置方案
候选长度即每轮由草稿模型提出的候选 token 数，代码中以参数 `candidate_length` 或实验变量 $ k $ 表示。候选长度设置直接影响推测解码的效率与风险平衡：当 $ k $ 较小时，目标模型每轮并行验证的收益有限，但候选失配带来的回退代价较小；当 $ k $ 较大时，若接受率较高，则单轮可推进更多 token，从而有效摊薄目标模型调用成本；但若候选后半部分频繁被拒绝，则会引入额外验证开销和缓存重建开销，反而削弱加速效果。

基于上述考虑，本文在实验程序中将候选长度设置为多个离散值，并采用统一脚本批量测试。当前代码默认使用 $ k \in \{1,2,4,6,8\} $ 作为候选长度集合。该设置既包含较短候选块，也覆盖中等长度候选块，能够较为直观地观察候选长度变化对接受率、平均接受前缀长度、端到端时间和吞吐率的影响。后续实验分析将围绕不同 $ k $ 下的综合表现展开。

### 3.5 模型配对方案设计
#### 3.5.1 同系列不同规模模型配对
为了体现一级异构推测解码中“轻量草稿模型 + 较强目标模型”的基本思想，本文优先考虑同系列、不同规模的模型配对方式。当前实现中通过 `MODEL_PAIR_PRESETS` 预设了多组草稿模型与目标模型组合，包括 `Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B`、`distilgpt2 -> gpt2` 以及 `gpt2 -> gpt2-medium`。其中，同系列不同规模模型通常共享相近的 tokenizer 和建模风格，有利于降低由于词表或架构差异引入的额外干扰，更适合作为一级异构推测解码的基础实验对象。

从方法设计角度看，这类配对具有两点优势。其一，小模型相较大模型前向推理更快，能够承担草稿候选生成的低成本职责；其二，同系列模型在分布形态上往往更接近，有助于提高候选接受率，从而更充分发挥推测解码的效率优势。

#### 3.5.2 不同模型配对的比较思路
在不同模型配对的比较中，本文主要关注以下几个方面。第一，模型间能力差距对接受率和平均接受前缀长度的影响。一般而言，草稿模型越接近目标模型，其局部预测越可能被接受，但草稿模型本身计算成本也可能上升。第二，模型规模差异对系统总耗时的影响。若草稿模型过小，虽然提议速度快，但可能因接受率偏低导致验证和回退开销上升；若草稿模型过大，则可能削弱“快速提议”带来的边际收益。第三，不同模型配对在不同解码策略下的适应性差异，即同一配对在贪心、Top-k 和 Top-p 条件下是否表现出一致的效率趋势。

因此，模型配对实验的核心不是单纯比较某一组模型是否更快，而是分析草稿模型计算成本、分布接近程度和最终系统收益之间的关系。这也是后续实验结果分析的重要切入点。

### 3.6 实现细节说明
#### 3.6.1 核心模块设计
本文的实现采用模块化设计，主要包括如下几个核心文件。

+ `common.py`：定义运行时配置 `RuntimeConfig`、采样配置 `SamplingConfig`、模型封装 `ModelBundle`，并提供模型配对预设、缓存目录解析、KV cache 转换、缓存裁剪、缓存重建以及 CSV 写出等通用功能。
+ `spec_decode.py`：实现一级推测解码主流程，是系统的核心模块。其中 `propose_block` 负责草稿候选生成，`verify_block` 负责目标模型并行验证，`corrective_distribution` 负责拒绝时的修正分布构造，`speculative_generate` 负责完整生成循环与统计信息汇总。
+ `baseline_generate.py`：实现仅使用目标模型的 baseline 生成流程。当前同时支持基于 Hugging Face `generate()` 的 `hf` baseline，以及采用手写自回归循环的 `manual` baseline，并在相同采样策略下输出生成文本、生成时间和吞吐率。
+ `run_experiment.py`：用于单次实验或单 prompt 多候选长度实验，适合快速检查某一组参数配置下的运行效果。
+ `run_bench.py`：用于批量 benchmark，是实验组织的主要入口，支持多 prompt、多解码策略、多候选长度、多重复轮次以及多种 baseline 实现的系统测试，并输出原始结果表与汇总结果表。

在缓存管理方面，本文实现兼容 Hugging Face 新旧两类缓存格式。如果环境中支持 `DynamicCache`，系统会将传统 `past_key_values` 转换为动态缓存对象，以提升兼容性；同时通过缓存裁剪与局部状态推进相结合的方式，在保证正确性的前提下降低拒绝回退带来的额外开销。该设计保证了代码在不同模型和不同 transformers 版本下具有较好的适配能力。

#### 3.6.2 baseline 对照实现
为了客观评估一级推测解码的收益，本文实现了两类 baseline 对照方案。第一类 baseline 仅保留目标模型，由目标模型调用 Hugging Face 提供的 `generate` 接口直接完成自回归生成，用于反映工程环境下较成熟生成快路径的性能表现。第二类 baseline 同样仅保留目标模型，但采用手写自回归循环逐 token 生成，并显式维护 KV cache、采样分布和状态推进过程。该实现与推测解码主流程在控制逻辑上更为接近，更适合进行同口径对比。

在对照设计上，两类 baseline 与推测解码共享相同 prompt、相同采样方式、相同输出长度上限和相同随机种子生成规则。这样可以尽量减少无关变量干扰，使吞吐率、总耗时以及输出一致性等指标的差异更能反映推测解码机制本身带来的影响。其中，手写自回归 baseline 尤其适合用于说明“算法收益”与“框架快路径收益”之间的差异，而 `generate()` baseline 则更接近实际应用中的工程性能参考。

#### 3.6.3 实验流程说明
本文实验的基本流程如下。首先，根据命令行参数确定模型配对、采样策略、候选长度集合、prompt 集合、重复次数、预热次数以及 baseline 实现类型，并完成模型加载。其次，对每个 prompt 和每种采样策略，分别运行 `hf` baseline 与 `manual` baseline 多次，将输出文本和性能数据记录下来；再对每个候选长度运行推测解码，并将结果与同轮次 baseline 输出做对比，计算输出精确匹配、前缀匹配长度和前缀匹配比例等指标。当前实现中，推测解码结果默认优先与手写自回归 baseline 的生成 token 序列进行对比，以减少不同工程路径造成的评测偏差。最后，将所有原始记录写入 `bench_raw.csv`，并按 prompt 维度与总体维度生成 `bench_summary.csv` 以供后续分析。

其中，预热轮次用于减少首次加载或首次执行带来的偶然开销对统计结果的影响；重复运行用于获得均值与标准差，使实验结论更稳健。整体实验流程具有较强的可复现性和批量扩展能力。

### 3.7 性能评价指标
#### 3.7.1 Token 接受率
Token 接受率用于衡量草稿模型提出的候选 token 中有多少比例最终被目标模型接受。设候选总 token 数为 $ N_p $，被接受的 token 数为 $ N_a $，则接受率定义为

$ \text{Acceptance Rate} = \frac{N_a}{N_p}. $

该指标反映草稿模型与目标模型在局部预测上的一致程度，是影响推测解码加速效果的核心因素之一。接受率越高，说明候选块中越多 token 可以直接复用，目标模型并行验证的收益越容易体现。

#### 3.7.2 平均接受前缀长度
平均接受前缀长度用于刻画每轮验证中，候选序列从起始位置开始连续被接受的 token 数的平均值。设总验证轮数为 $ R $，第 $ r $ 轮被接受的连续前缀长度为 $ L_r $，则有

$ \bar{L} = \frac{1}{R}\sum_{r=1}^{R}L_r. $

与总体接受率相比，该指标更能反映“候选块能否在前部形成稳定推进”的情况。即使总体接受率接近，如果接受 token 总是集中在靠后位置，该指标也可能较低。因此，它适合与接受率结合使用，刻画接受行为的细粒度差异。

#### 3.7.3 端到端生成时间
端到端生成时间指从开始生成到达到设定输出长度上限为止的总耗时。该指标直接反映用户实际感知的生成延迟，是评价加速方法是否具有工程价值的重要依据。本文实现中在正式生成开始前后记录时间戳，并在 GPU 环境下通过同步操作降低异步执行对计时结果的影响。

#### 3.7.4 吞吐率
吞吐率用于描述单位时间内生成 token 的数量，定义为

$ \text{Tokens/s} = \frac{\text{Generated Tokens}}{\text{Total Generation Time}} $

吞吐率与端到端时间互为补充。对于固定输出长度任务，端到端时间越短，吞吐率通常越高。本文在结果分析中将吞吐率作为衡量不同配置效率差异的主要指标之一。

#### 3.7.5 开销拆解相关指标
除了总体效率指标外，本文还在实现中额外记录了草稿候选生成时间、目标并行验证时间以及拒绝后的缓存恢复时间，分别对应 `draft_time`、`verify_time` 和 `rebuild_time`。同时，还统计了三者相对于总生成时间的占比指标。为使这些分项统计更准确，当前实现对各阶段的模型前向调用采用了同步计时口径，使 GPU 异步执行带来的阶段间时间串扰尽可能减小。上述指标可用于分析系统时间主要消耗在哪一阶段，从而识别性能瓶颈。例如，当候选长度过大且接受率偏低时，缓存恢复时间可能显著上升；当草稿模型本身较大或采样分布处理代价较高时，候选生成时间也可能成为限制因素。

此外，系统还记录了候选总数、接受 token 数、拒绝 token 数和验证轮数等中间统计量，这些数据能够为后续开销拆解与行为分析提供支撑。

### 4.2 测试任务与实验分组
#### 4.2.1 Prompt 设置与任务类型划分
为保证实验结果具有一定代表性，本文对测试输入的 prompt 进行了适度设计。相关研究表明，提示词的表达形式和任务类型会对大语言模型的生成结果产生影响，不同类型的 prompt 可能在输出长度、语义组织方式以及生成稳定性等方面表现出差异。因此，在推测解码实验中，仅采用单一类型的测试输入，可能会使实验结果受到特定任务形式的干扰，难以全面反映不同解码策略的实际表现。故本文从概念解释、对比分析、机制说明和原因分析四类任务中各选取一个 prompt，用于比较不同解码方式和不同候选长度条件下的推测解码表现。具体如下：

（1）概念解释类  
`Explain speculative decoding in simple terms.`

（2）对比分析类  
`Compare greedy decoding, top-k sampling, and top-p sampling in one paragraph.`

（3）机制说明类  
`Explain how draft models and target models interact during speculative decoding.`

（4）原因分析类  
`Why can a low acceptance rate reduce the speedup of speculative decoding?`

本文设置上述 prompt 的目的在于覆盖不同常见生成形式，并在资源受限条件下尽量保证实验结果的稳定性与可比性。

#### 4.2.2 不同解码方式对比实验
不同解码方式对比实验的目的是考察推测解码在确定性生成与随机采样生成条件下的表现差异。具体做法是在相同模型配对、相同 prompt 集和相同最大输出长度条件下，分别采用贪心解码、Top-k 采样和 Top-p 采样进行实验，并比较各自的接受率、平均接受前缀长度、端到端生成时间和吞吐率。与此同时，还将推测解码结果分别与 `hf` baseline 和 `manual` baseline 进行对照，以区分算法因素与工程实现因素。

该实验重点回答两个问题：其一，不同采样策略是否会显著改变草稿模型与目标模型之间的接受行为；其二，推测解码的加速效果是否在随机采样条件下依然成立。

#### 4.2.3 不同候选长度对比实验
不同候选长度对比实验的目的是分析候选块长度 $ k $ 对推测解码性能的影响。实验中在固定模型配对和采样策略的前提下，依次测试多个候选长度，并考察随着 $ k $ 增大，接受率、平均接受前缀长度、总耗时和吞吐率如何变化。

该实验用于寻找候选长度设置中的效率平衡点，即在并行验证收益与拒绝回退代价之间取得较优折中。

#### 4.2.4 不同模型配对对比实验
不同模型配对对比实验旨在比较多组“草稿模型 + 目标模型”组合在推测解码中的实际表现。实验将统一 prompt 集、候选长度集合和采样策略，对各组模型配对分别进行 benchmark，并分析模型间规模差异、同系列一致性与草稿建模能力对最终性能的影响。

该实验有助于揭示：并非任意小模型都适合作为草稿模型，只有当“提议速度”和“分布接近度”之间达到较好平衡时，推测解码才能体现优势。

#### 4.2.5 开销拆解实验
前文分别从解码方式、候选长度和模型配对三个角度分析了一级推测解码的整体表现，但仅凭吞吐率、接受率和平均接受前缀长度等指标，仍不足以准确说明不同配置下性能差异究竟来自哪里。而开销拆解实验的目标是分析端到端生成时间由哪些部分构成，进一步将端到端生成时间拆分为草稿生成、目标验证和状态恢复等多个阶段，分析不同实验条件下系统主要瓶颈的具体来源。

结合实现中记录的 `draft_time`、`verify_time`、`rebuild_time` 及其占比指标，以及验证轮数、拒绝 token 数等统计量，可进一步评估草稿生成、目标验证和缓存恢复各自占比，并据此定位系统瓶颈。

该实验特别适合用于解释以下现象：在某些配置下，即使接受率并不算低，端到端加速比仍可能有限；或者在某些较大候选长度设置下，吞吐率不升反降。

### 4.3 不同解码方式下的性能分析
从整体结果看，贪心解码、Top-k 采样和 Top-p 采样在一级推测解码中的表现存在明显差异，而且这种差异在三组模型配对上基本保持一致。对于 `distilgpt2 -> gpt2`，贪心解码在所有候选长度下都优于采样路径，最佳吞吐率出现在 `k=6`，平均达到约 60.21 tokens/s，且输出与 `manual` baseline 完全一致。对于 `gpt2 -> gpt2-medium`，贪心解码同样最稳定，最佳点出现在 `k=4` 附近，平均吞吐率约为 34.29 tokens/s。对于 `Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B`，三种解码方式整体都较慢，但贪心解码仍优于两种采样路径，最佳点出现在 `k=2`，平均吞吐率约为 8.50 tokens/s。

出现上述现象的根本原因在于，不同解码方式直接改变了草稿模型与目标模型之间的局部一致性。贪心解码仅要求两者在 argmax 位置保持一致，因此整体接受率最高。例如在 `distilgpt2 -> gpt2` 中，贪心路径从 `k=1` 到 `k=8` 的平均接受率大多维持在 0.57 以上；在 `gpt2 -> gpt2-medium` 中，贪心路径在 `k=4` 时仍能保持约 0.76 的平均接受率。相比之下，Top-k 和 Top-p 都引入了随机采样，候选前缀更容易在较早位置发生失配，接受率和输出一致性均明显下降。

随机采样对输出一致性的影响也十分明显。以 `manual` baseline 为参考，贪心路径在前两组 GPT-2 系列实验中都实现了完整输出一致；而在采样条件下，`output_exact_match_mean` 基本降为 0，说明即使整体吞吐率仍有一定提升，生成结果也会偏离单纯目标模型路径。这并不意味着算法失效，而是说明在随机采样场景下，推测解码更适合从接受率、平均接受前缀长度和端到端效率角度进行评价，而不应再把逐序列完全一致作为核心目标。

### 4.4 不同候选长度下的性能分析
#### 4.4.1 不同候选长度下的整体性能变化  
从实验结果来看，三组模型配对在候选长度变化下均表现出较为一致的总体趋势：随着 $k$ 从较小值逐步增大，系统吞吐率通常先上升后下降，说明候选长度并非越大越好，而是存在相对合适的取值区间。其原因在于，当 $k$ 较小时，目标模型一次验证只能覆盖较短候选块，并行优势尚未充分释放；而当 $k$ 继续增大后，虽然单轮理论推进能力增强，但草稿模型额外 proposal 成本和候选尾部失配带来的恢复代价也会同步累积，最终使端到端效率出现回落。  

在 `distilgpt2 -> gpt2` 中，这一趋势最为典型。贪心条件下吞吐率从 `k=1` 的约 44.72 tokens/s 上升到 `k=6` 的约 60.21 tokens/s，但继续增大到 `k=8` 后回落到约 56.20 tokens/s。对应地，平均接受前缀长度从约 0.88 增长到约 4.51，而接受率则从约 0.88 下降到约 0.57。这说明更长候选块确实带来了更大的单轮推进能力，但也使候选块后部更容易失配，因此收益并不会随着候选长度增加而线性提升。

`gpt2 -> gpt2-medium` 的趋势与之相似，但最优区间更靠前。在贪心条件下，`k=4` 的平均吞吐率约为 34.29 tokens/s，已略高于 `k=6` 的约 34.15 tokens/s，而 `k=8` 明显下降到约 31.69 tokens/s。这说明当目标模型相对更重、草稿模型也不再足够轻量时，过大的候选长度会更快放大草稿侧负担。

`Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 则进一步表明，并非更大的 $k$ 一定更优；该组在贪心 `k=2` 时达到最高吞吐率约 8.50 tokens/s，而 `k=4` 之后便持续下降，说明在当前配对下，候选长度稍一增大，新增 proposal 成本就会较快压过并行验证带来的收益。

综合来看，不同模型配对虽然最优 $k$ 的具体位置不同，但都说明一级推测解码中的候选长度存在明显的收益边界。过小的候选长度会抑制并行验证优势，过大的候选长度则会使草稿生成和后续恢复开销迅速累积，因此更合理的策略是在中等候选长度区间内寻找性能平衡点，而不是简单追求更长的候选块。  

【插图建议：可在此处配“候选长度与吞吐率关系图”，建议三组模型各一条曲线，横轴为 $k$，纵轴为 tokens/s，用于直观体现先升后降趋势。】

#### 4.4.2 平均接受前缀长度与候选长度的关系
仅观察整体接受率，还不足以完整解释不同候选长度下的性能差异。相比之下，平均接受前缀长度能够更细致地刻画每轮验证中，候选块前部究竟能够稳定推进多远，因此更适合用于解释候选长度变化背后的性能机制。

实验结果表明，随着候选长度增大，平均接受前缀长度通常会同步上升，但其增长速度并不总能弥补候选块整体变长所带来的额外成本。以 `distilgpt2 -> gpt2` 的贪心结果为例，随着 $k$ 增大，平均接受前缀长度由约 0.88 提升至约 4.51，说明目标模型在更长候选块上确实能够一次接受更多前部 token，单轮推进能力明显增强；但与此同时，整体接受率却由约 0.88 下降至约 0.57，表明候选块后部位置更容易发生失配。也就是说，更大的 $k$ 虽然提高了“每轮能接受多少”，却也降低了“候选块整体有多稳定”。

这一现象说明，平均接受前缀长度与整体接受率虽然都能反映接受行为，但二者关注的侧重点并不相同。前者刻画的是候选块前部连续推进能力，后者反映的是候选 token 的总体复用比例。在候选长度较短时，两者变化通常较为一致；而在候选长度较大时，候选前部可能仍能保持较好接受效果，但后部失配概率显著上升，于是就会出现“平均接受前缀长度增加，而整体接受率下降”的情况。此时，如果只看接受率，容易低估较长候选块在前段推进上的实际作用；如果只看平均接受前缀长度，则又容易忽略尾部失配对总成本的拖累。因此，二者需要结合使用，才能更全面地解释候选长度变化的真实影响。

对于 `gpt2 -> gpt2-medium` 和 `Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 两组配对，也可以观察到类似规律。即便在某些中等 $k$ 取值下，平均接受前缀长度仍然有所提升，但最终吞吐率并没有同步改善。这进一步说明，平均接受前缀长度更适合作为解释性指标，用来揭示候选长度为何在某一范围内有效，又为何在更大范围内开始失效；至于造成收益停止增长的更深层原因，则需要结合后文的模型配对分析和开销拆解一起理解。

【插图建议：可在此处配“平均接受前缀长度随候选长度变化图”，与上一图形成呼应，突出‘接受前缀变长但收益未必同步增加’的现象。】

#### 4.4.3 候选长度的收益边界分析  
结合吞吐率、整体接受率和平均接受前缀长度三个维度可以看出，候选长度的作用并不是单向增益，而是存在明确的收益边界。对于一级推测解码而言，候选长度带来的正向作用主要体现在：目标模型能够一次验证更多位置，从而摊薄逐 token 解码成本；而其负向作用则体现在：草稿模型需要额外提出更长候选序列，候选块后部失配概率更高，状态恢复与缓存维护压力也随之增加。

因此，候选长度优化的关键不在于单纯提高 $k$，而在于找到“接受前缀增长速度”和“额外开销增长速度”之间的平衡点。就本文的实验结果而言，`distilgpt2 -> gpt2` 更适合取 `k=4∼6` 的中等偏大区间，`gpt2 -> gpt2-medium` 的较优区间更接近 `k=4`，而 `Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 则更适合较小候选长度。这些差异本质上反映了不同模型配对下 proposal 成本与分布接近度之间的不同平衡关系。

综上，候选长度分析表明：一级推测解码并不存在统一最优的通用 $k$，其最佳取值依赖于具体模型配对和实现条件；但从整体规律看，中等候选长度往往比极端的小 $k$ 或大 $k$ 更容易取得较好的端到端收益。这一结论也为后续模型配对分析和开销拆解分析提供了重要基础。

### 4.5 不同模型配对下的性能分析
#### 4.5.1 不同模型配对的整体性能比较  
三组模型配对结果表明，推测解码收益并不只取决于“草稿模型比目标模型更小”，而是同时取决于草稿提议成本和草稿-目标分布接近度。`distilgpt2 -> gpt2` 是当前三组中表现最好的组合：在贪心条件下，`k=6` 达到约 60.21 tokens/s，明显高于另外两组；同时其接受率和平均接受前缀长度在多数配置下也保持较高水平。这说明 `distilgpt2` 既足够轻量，又与 `gpt2` 保持了较好的局部预测一致性，因此能够较充分地体现一级推测解码的潜在优势。

`gpt2 -> gpt2-medium` 则呈现出“接受率尚可但整体收益有限”的典型特征。该组在贪心条件下也能维持较高接受率，例如 `k=4` 的平均接受率约为 0.76，但最佳吞吐率只有约 34.29 tokens/s，明显弱于 `distilgpt2 -> gpt2`。这说明该组合虽然在局部一致性上并不存在明显问题，但草稿模型 `gpt2` 本身已经不够轻量，使其难以像更轻量的草稿模型那样释放明显优势。

`Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 的情况更加明显。虽然两者属于同一模型系列，理论上应具有一定分布相关性，但实验结果显示该组合整体吞吐率偏低，且在候选长度稍大时就出现明显回落。这说明在当前硬件与实现条件下，即便同系列模型具备一定分布优势，若草稿模型前向本身仍较重，则一级推测解码依然难以表现出理想收益。

总体而言，三组结果共同表明：模型配对优劣不能简单用“参数量更小”来判断，而应同时考察草稿模型是否足够便宜、是否与目标模型足够接近，以及这两者是否能在整体性能上形成正向合力。

【插图建议：可在此处配“不同模型配对整体性能比较图”，建议使用柱状图比较三组配对在最优贪心配置下的吞吐率。】

#### 4.5.2 不同模型配对的接受行为差异
从接受行为角度看，不同模型配对之间的差异进一步解释了其整体性能分层。`distilgpt2 -> gpt2` 在贪心条件下不仅整体接受率较高，而且平均接受前缀长度也较为理想，说明该组合能够较稳定地在候选块前部形成连续推进。这种“接受率较高且接受前缀较长”的特征，使得目标模型一次验证常常能够有效覆盖多个 token，从而更容易把并行验证优势转化为可见的性能改善。

`gpt2 -> gpt2-medium` 则表现出另一种更常见的情况，即接受行为本身并不差，但依然不足以支撑明显收益。该组在某些配置下仍可维持较高接受率和相对较长的接受前缀，说明局部预测一致性并非主要瓶颈；但较好的接受行为并没有同步转化为更高的整体性能。这表明在一级推测解码中，“接受得够多”只是必要条件，而不是充分条件。

`Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 则同时暴露出接受行为和整体效率两方面的限制。一方面，该组在贪心条件下的输出一致性不如前两组稳定，说明即便在确定性路径下，其局部一致性也未达到非常理想的程度；另一方面，随着候选长度增大，吞吐率并未表现出可持续改善趋势。因此，该组更适合作为一级推测解码适用边界的例子来分析，而不是作为理想有效配对来展示。

由此可见，接受行为差异固然重要，但只有当它与较低的草稿提议成本同时出现时，才可能真正形成有效的模型配对。否则，即使接受率看起来并不低，整体性能也未必足以体现推测解码的优势。

#### 4.5.3 与 baseline 的对照分析
为了更准确地判断不同模型配对的实际价值，还需要将一级推测解码结果与 baseline 进行对照分析。与 `manual` baseline 的比较表明，在本文三组模型配对中，只有 `distilgpt2 -> gpt2` 在贪心条件下能够明显接近 baseline，但即便如此，其总体平均吞吐率仍未超过纯目标模型自回归生成。

具体而言，该组 `manual` baseline 的贪心吞吐率约为 71.79 tokens/s，而一级推测解码在 `k=6` 时达到最佳值约 60.21 tokens/s。这说明该组已经较充分释放了一级推测解码的并行验证收益，但草稿侧额外前向开销仍然限制了其进一步逼近甚至超过 baseline。

`gpt2 -> gpt2-medium` 的情况更能说明“接受率尚可不等于真实加速”。该组最佳贪心结果约为 34.29 tokens/s，而其 `manual` baseline 约为 40.30 tokens/s，说明当前一级推测解码尚未在端到端层面形成优势。

至于 `Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 组，其最佳贪心吞吐率约为 8.50 tokens/s，也明显低于约 16.36 tokens/s 的 baseline，进一步表明在当前条件下该组配对并不适合作为高收益一级推测解码方案。

从输出一致性看，前两组 GPT-2 系列实验中的贪心路径与 `manual` baseline 保持了完整一致，说明在确定性生成条件下，当前一级推测解码实现并未改变最终输出结果；而在 Top-k 和 Top-p 条件下，`output_exact_match_mean` 基本降为 0，这与随机采样本身的非确定性相一致。需要特别指出的是，`Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 的贪心结果在 `k≥2` 时并未始终保持完全一致，部分配置下 `output_exact_match_mean` 仅为 0.25 到 0.50，前缀匹配比例也明显低于前两组 GPT-2 系列实验。这说明该组在当前实现与配置下的贪心一致性仍不够稳定，更适合作为性能边界与适用局限来讨论；若后续继续扩展该配对，相关结果还需要结合实现细节与数据记录过程进一步复核。

因此，baseline 对照分析进一步说明：不同模型配对的优劣不能仅由接受率或局部推进能力判断，还必须结合基准生成路径下的端到端吞吐率与输出一致性综合评估。只有当草稿模型额外 proposal 成本足够低，且其与目标模型保持较高分布一致性时，一级推测解码才可能在真实系统中体现出稳定收益。与此同时，本节的比较也为下一节的开销拆解提供了分析前提：模型配对差异不仅体现为“能否接受”，也体现为“接受之后是否值得”。

### 4.6 性能开销拆解
#### 4.6.1 草稿生成、目标验证与状态恢复的时间构成
结合同步计时后的 `draft_time`、`verify_time` 与 `rebuild_time` 统计结果可以看到，当前实现下一级推测解码的主要瓶颈已经明显从“拒绝后的恢复成本”转移到了“草稿模型提议成本”。

以 `distilgpt2 -> gpt2` 的贪心路径为例，`k=1` 时 `rebuild_time_ratio` 仍接近 0.48，而 `draft_time_ratio` 仅约为 0.18；但随着 $ k $ 增大到 6 和 8，`draft_time_ratio` 分别上升到约 0.53 和 0.60，成为端到端时间中的最大组成部分，而 `rebuild_time_ratio` 已下降到约 0.24 和 0.17。说明在完成缓存裁剪优化与末轮无效 rebuild 跳过后，恢复阶段已不再是主要矛盾。

这一趋势在另外两组模型配对中更加明显。`gpt2 -> gpt2-medium` 的贪心路径下，`draft_time_ratio` 从 `k=1` 时的约 0.30 持续升至 `k=8` 时的约 0.60；`Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 中，该比例在贪心 `k=8` 时甚至达到约 0.71。相比之下，`verify_time_ratio` 通常随着候选长度增加而下降到约 0.10 至 0.20 区间，说明并行验证带来的收益确实存在；`rebuild_time_ratio` 则大多稳定在约 0.17 至 0.30 范围内，虽然仍不可忽略，但已不再决定整体性能上限。

#### 4.6.2 候选接受位置与性能瓶颈的关系
除了分阶段时间占比之外，候选接受行为本身也会进一步影响系统瓶颈的迁移。特别是候选首次拒绝出现的位置，会直接决定一轮验证中到底有多少 token 能够被真正复用，以及状态恢复开销是否值得支付。若候选块前部即可稳定接受较长前缀，则目标模型一次验证可以覆盖多个有效位置，验证收益较容易体现；若候选在较早位置就频繁失配，则草稿侧额外 proposal 成本难以摊薄，系统就更容易表现为“多做了候选生成，却没有换来足够多的有效推进”。这一点也正是平均接受前缀长度能够解释候选长度收益边界的重要原因。

从本文结果看，`distilgpt2 -> gpt2` 在贪心条件下之所以能够成为最优配对之一，不仅是因为整体接受率较高，更因为其候选块前部通常能够保持较稳定的连续接受，从而让目标模型的块级验证真正发挥作用。相反，在 `gpt2 -> gpt2-medium` 和 `Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 中，即便某些配置下接受率并不算很低，但由于草稿模型 proposal 成本较高，或者候选有效接受位置不够靠后，最终仍无法形成足够强的端到端收益。换言之，性能瓶颈并不只取决于“接受了多少”，还取决于“在哪些位置接受”以及“这些接受是否足以覆盖草稿额外成本”。

因此，将候选接受位置与开销拆解结合起来看，可以更准确地理解一级推测解码中的收益形成机制：只有当候选前部能稳定形成较长接受前缀时，目标模型验证收益才有可能覆盖草稿 proposal 和少量恢复开销；否则，系统很快就会进入“草稿生成越多，浪费越多”的状态。

#### 4.6.3 不同配置下的主要开销来源总结
总之，当前代码版本下的主要性能瓶颈可以概括为：随着候选长度增加，目标模型验证被更多地并行摊薄，但草稿模型逐 token 提议的累计成本迅速上升，并最终主导端到端耗时，说明在不改变论文实验主线的前提下，后续更值得关注的不是继续压缩 `rebuild_time`，而是如何在固定策略下解释并控制 `draft_time` 的主导效应。随着候选长度增加，`rebuild_time` 虽然仍然存在，但通常不是决定性能上限的核心因素；相反，`draft_time` 是否能够保持足够低，直接决定了更长候选块能否真正转化为有效收益。

这一结论能够很好地解释前文中的多个现象。首先，它解释了为何最优候选长度往往出现在中等区间，而不是一味向更大 k 延伸：因为当 k 增大到一定程度后，`draft_time` 的增长会快于目标模型验证收益的增长。其次，它解释了为何 `distilgpt2 -> gpt2` 的整体表现优于另外两组：该组合的草稿模型更轻量，因此更容易把并行验证优势保留下来。再次，它也说明了为何 `gpt2 -> gpt2-medium` 和 `Qwen/Qwen2.5-0.5B -> Qwen/Qwen2.5-1.5B` 更适合作为一级推测解码的边界案例来讨论：它们的问题并不主要是恢复阶段太慢，而是草稿侧本身已不够“便宜”。

因此，从性能开销拆解角度看，后续若要进一步优化一级异构推测解码，重点不应只放在减少拒绝后的恢复代价上，更应着重降低草稿 proposal 阶段的总体成本，并提高候选前部的稳定接受能力。这一结论不仅为后续工程优化提供了明确方向，也为多层级推测解码的进一步扩展提供了重要依据。

### 4.7 多层级推测解码初步结果与讨论
在前文已经完成不同解码方式、候选长度、模型配对、接受行为以及开销拆解分析的基础上，本节进一步将讨论视角从“一级结构是否有效”延伸到“多层结构是否值得继续扩展”。本文当前实现与实验重点集中在一级推测解码，因此尚未给出完整的多层级推测解码实测 benchmark 结果；不过，从现有一级结果出发，仍可以对多层级扩展方向作出初步讨论。一级实验已经表明，随着目标模型验证频率下降，系统瓶颈会逐步从验证侧转移到草稿提议侧；这意味着多层级结构虽然有潜力继续减少最重目标模型的调用次数，但也很可能同时累积更多前级草稿模型的生成开销，并引入更复杂的状态维护与接受链条。

结合第 4.3 至第 4.6 节的结果可以推断，多层级结构是否有效，取决于相邻层之间是否同时满足三个条件：其一，上层目标模型足够重，因而减少其调用次数确实有价值；其二，下层草稿模型足够轻，不会把新增 proposal 成本迅速放大；其三，相邻两层模型的条件分布足够接近，使得层间接受率不会随着层数增加而快速恶化。若这三个条件不能同时成立，那么多加一层并不一定带来端到端收益，反而可能因为 proposal 链变长、缓存维护更复杂以及误差累积更明显而降低整体性能。

因此，多层级推测解码未必会线性放大一级方法的收益。若各层之间的模型配对不能同时满足“上层足够重、下层足够轻、相邻层分布又足够接近”这三个条件，那么额外增加层级很可能只是在前端堆叠更多 proposal 成本，而无法等比例换来验证收益。在这一意义上，一级实验的价值不仅在于给出当前可复现的性能结果，也在于为后续多层级设计提供了筛选标准：未来扩展时应重点关注层间配对质量、候选长度累积效应以及瓶颈迁移问题，而不能只关注理论上的目标模型调用次数减少。

因此，本文将多层级 speculative decoding 视为后续工作方向，而不是本章的实证重点。一级实验已经为这一扩展提供了较明确的筛选依据：未来若要继续向多层级结构推进，应优先选择“草稿足够便宜且与上层模型分布足够接近”的模型对，并重点考察层数增加后 `draft_time` 是否继续主导总时间、层间接受率是否加速衰减，以及新增层级是否真正降低了最重目标模型的有效调用成本。由此，第四章的分析逻辑也在此处完成收束：一级实验不仅给出了当前实现下的性能结论，也为第五章关于研究结论、局限性与后续展望的展开提供了直接依据。

# 第5章 总结与展望
### 5.1 全文工作总结
本文围绕自回归大模型推理效率优化问题，重点研究了一级异构推测解码的设计、实现与实验分析。在理论部分，本文梳理了大语言模型的自回归生成机制、常见解码方法以及推测解码的基本原理，分析了草稿模型与目标模型协同生成、候选接受与拒绝回退等关键机制。在方法实现部分，本文设计并实现了一个可复现的一级异构推测解码实验框架，支持不同模型配对、不同解码方式和不同候选长度条件下的批量测试。在实验部分，本文从解码方式、候选长度、模型配对和开销拆解等角度，对推测解码的性能表现进行了系统分析，并在此基础上对多层级异构推测解码的扩展方向进行了初步讨论。  

### 5.2 主要研究结论与不足
通过实验分析，本文得到以下几点结论。  
第一，推测解码的实际收益与草稿模型和目标模型之间的匹配关系密切相关。只有当草稿模型既具有较低提议成本，又能与目标模型保持较高局部一致性时，推测解码才更可能体现出端到端性能优势。  
第二，不同解码方式会显著影响推测解码的接受行为与性能表现。总体来看，贪心解码通常更容易获得较高接受率和更稳定的性能，而采样策略虽然更贴近开放式生成场景，但也更容易引入候选失配。  
第三，候选长度并非越大越好，而是在并行验证收益与候选失配风险之间存在一个较优区间。候选长度过小会限制目标模型并行验证的收益，过大则可能带来额外的草稿生成与回退恢复开销。  
第四，端到端性能不仅取决于接受率本身，还受到草稿生成时间、目标验证时间和缓存恢复时间等系统开销的共同影响，因此仅从理论接受行为出发并不足以完全判断推测解码的真实收益。

尽管本文取得了一定结果，但仍存在一些不足。首先，本文实验主要围绕有限规模的模型组合展开，模型范围和任务规模仍有进一步扩展空间。其次，本文的实证研究重点集中在一级异构推测解码，对多层级异构推测解码尚未开展完整系统的实验验证。最后，本文实验主要在资源受限的本地单机环境下进行，所得结论在更大规模模型和更高性能硬件平台上的适用性，仍需进一步验证。

### 5.3 未来研究展望
后续研究可以从以下几个方面展开。其一，可进一步扩展模型配对范围，引入更多同系列或跨系列模型组合，分析不同模型关系对推测解码性能的影响。其二，可围绕多层级异构推测解码开展更系统的研究，重点考察层间接受率、额外 proposal 成本以及缓存维护复杂度对整体收益的影响。其三，可结合更多任务类型和更丰富的测试数据，对推测解码在不同文本生成场景下的稳定性进行评估。其四，在工程实现层面，还可以继续优化缓存裁剪、状态恢复和候选生成流程，以进一步提升推测解码在真实部署环境中的实用价值。
